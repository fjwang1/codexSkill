# 单人文章讲解视频稿

## 目标

把一个本地文章项目改写成中文单人解释型视频口播稿。最终稿要帮助没有读过原文的观众听懂：这件事是什么，为什么会这样，它背后的机制是什么，文章真正想提醒我们的判断是什么。

不要模仿任何创作者的口头禅、人设或标志性表达。只借用稳定的方法：把文章讲成一个可理解的解释模型，而不是段落摘要。

默认判断：每日外刊筛选出来的文章，大多数是解释型文章。它们通常在回答“为什么发生”“这个现象说明了什么”“为什么直觉理解不够”。除非文章明显是人物特写、访谈、纯时间线或叙事报道，否则优先使用“默认解释型主模板”。

## 输入

必需文件：

```text
<project>/source/article.txt
```

优先读取的可选文件：

```text
<project>/source/source_metadata.json
<project>/source/reading_doc.md
<project>/planning/article_brief.json
<project>/planning/context_cards.json
<project>/planning/episode_outline.json
```

如果项目目录不是标准 `source/` 结构，就读取最接近的本地原文文件，并在报告里说明使用了哪个替代输入。

## 输出

```text
<project>/single_host_script.md
<project>/planning/single_host_explainer_plan.json
<project>/planning/single_host_script_generation_report.md
```

不要覆盖 `podcast_script.md`。这个 Skill 只负责单人口播解释稿，不负责双人播客。

## 核心原则

对大多数文章筛选项目，直接使用“默认解释型主模板”。不要在文章已经明显是解释型文章时，过度自由地另选风格。

每篇稿子都必须有：

- 外刊框架之前的中文观众生活场景，或者一个真实的生活张力
- 表面现象，以及它和旧经验、常识经验之间的反差
- 一个早期纠偏：这件事表面像是在讲 A，真正关键是 B
- 一个贯穿全文的主问题
- 当原文支持多原因解释时，在来源介绍之后的前 500 个汉字内明确说出 4 到 6 个核心变量
- 一组自洽的“提问，短答，机制解释，例子落地，转入下一问”
- 至少一个反直觉澄清
- 一个真实的限制、代价、矛盾或未解决问题
- 一个观众能记住的大判断，而不是简单复述总结

章节数量和顺序要服从原文材料，不要为了整齐而硬凑。

## 默认解释型主模板

把下面结构当作主路线，而不是填空表。

```text
1. 中文生活经验开场
   先从中国观众熟悉的经验说起，不要一上来列外来食物、外来概念或原文标题。
   例子：以前说吃得好，可能是大鱼大肉，请客有面子，年夜饭桌上得有几个硬菜。

2. 现象反差
   说明现在画风哪里变了。
   例子：但现在，很多人说吃得好，画风变了：三文鱼、蓝莓、牛油果、有机菜、低脂高蛋白、干净配料表。

3. 外刊命名
   只在有帮助时提一次来源给出的标签。
   例子：《经济学人》把这个变化叫作中国中产饮食的“加州化”。

4. 纠偏
   立刻纠正常见浅层理解。
   例子：但这不是中国人突然变洋气，也不是大家都爱上牛油果。

5. 主问题或核心判断
   说清楚原文真正解释的东西。
   例子：真正的问题是，当一个社会从吃得饱走向吃得好，食品产业、农业供应链、消费心理和治理压力会怎么一起变。

6. 核心变量组
   尽早明确说出 4 到 6 个变量。
   例子：这里面大概有五个变量：钱变多了，食品安全的不信任还在，健康焦虑上来了，供应链跟上了，大家又忙又懒，所以健康食品还必须方便。

7. 逐变量机制拆解
   正文每一节拆一个变量，或拆一组相互绑定的变量，并用原文事实支撑。

8. 约束和代价
   说明趋势为什么真实，但又为什么有限：价格、信任、时间、物流、政策、阶层、地域、不确定性。

9. 大判断收束
   结尾要留下一个能被观众记住的框架，不要只是复盘。
   例子：吃饭这件事，已经从温饱问题，变成了信任问题、健康问题、效率问题和阶层问题。
```

替换例子时，要换成原文对应的中文生活场景、变量、事实和判断。

## 工作流程

1. 读取本地文章和已有策划文件。
   - 如果有 `article_brief.source_facts`、术语表或 `do_not_invent`，把它们当作事实边界。
   - `context_cards` 只用于解释背景和定向，不要让背景卡片替代原文。
   - 如果已有双人播客的 `episode_outline`，可以复用事实和段落目的，但要把对话推进改成单人口播的解释推进。

2. 先判断文章类型。
   - 默认使用 `default_explainer` 作为 `primary_engine`。
   - 只有当文章明显不是解释型文章时，才选择其他引擎。

   可选引擎的含义如下：

   ```text
   default_explainer：默认解释型文章，先纠偏，再给变量组，再逐步拆机制。
   mechanism：解释某件事为什么会发生，重点是变量、激励、成本、赢家、输家、瓶颈。
   trend：解释一个社会、消费或经济趋势如何出现，重点是过去和现在、驱动因素、限制、未来方向。
   system：解释制度、供应链、政策或基础设施如何运转，重点是节点、流程、失败点、协调难度。
   misconception：观众容易以为是 A，但关键其实是 B，重点是纠正常见误读。
   conflict：多个主体目标冲突，重点是激励、筹码、限制、误判和后果。
   person-as-window：用一个人物或故事打开更大的行业、地方、代际或系统问题。
   case-to-structure：用一个具体案例揭示更大的结构性模式。
   ```

3. 把文章压缩成一个主问题和一组核心变量。
   - 主问题要具体、因果化，不要只是复述标题。
   - 好的主问题例子：

   ```text
   为什么中国中产饮食正在变得更健康、更轻、更像“加州生活方式”？
   为什么一个战场上的导弹防御成功，可能误导另一个战场的决策者？
   为什么一个看上去成功的政策，可能掩盖了底层系统的脆弱？
   ```

   - 当原文用多个原因解释一个现象时，建立 4 到 6 个变量。
   - 变量名称要自然，优先使用“钱”“信任”“健康焦虑”“供应链”“便利”“政策压力”这类中文词。
   - 不要为了对称硬凑。如果原文只支撑三个变量，就只写三个。

4. 生成 `planning/single_host_explainer_plan.json`。
   使用下面 schema。字段名保持英文，方便机器和下游脚本稳定读取；字段内容用中文写。

   ```json
   {
     "schema_version": "single-host-explainer-plan.v1",
     "source_title": "...",
     "source_publication_zh": "...",
     "primary_engine": "default_explainer | mechanism | trend | system | misconception | conflict | person-as-window | case-to-structure",
     "secondary_engines": ["..."],
     "main_question": "...",
     "viewer_initial_misread": "...",
     "native_life_hook": "...",
     "phenomenon_contrast": "...",
     "early_driver_set": ["..."],
     "explanation_model": {
       "surface_phenomenon": "...",
       "drivers": ["..."],
       "stakeholders": ["..."],
       "constraints": ["..."],
       "counterintuitive_point": "...",
       "why_now": "..."
     },
     "sections": [
       {
         "name": "自然中文小标题",
         "purpose": "这一节帮助观众理解什么",
         "source_facts": ["原文支撑的事实"],
         "explanation_moves": [
           "提出一个具体问题",
           "给出短答案",
           "用原文事实解释机制",
           "落到一个具体场景",
           "转入下一个问题"
         ],
         "avoid": ["避免夸大的说法或跑偏内容"]
       }
     ],
     "visual_chapter_notes": [
       {
         "section": "...",
         "visual_role": "number card | comparison table | flow diagram | map | timeline | object scene | quote card",
         "note": "PPT 这一页应该把什么东西可视化"
       }
     ]
   }
   ```

5. 写 `single_host_script.md`。
   使用下面格式：

   ```markdown
   # 单人视频讲解稿：<标题>

   来源文章：《<中文来源名>》<中文标题或自然中文题名>
   形式：单人中文解释型视频口播
   建议时长：<估计时长>

   ## 正文

   <正文口播稿>
   ```

   可以保留小标题帮助后续做 PPT，但正文必须是可以直接念出来的中文。不要出现说话人标签。

6. 写生成报告。
   报告包括：

   - 读取了哪些文件
   - 选择了哪个解释引擎，为什么
   - 使用了哪些原文或策划事实
   - 有哪些事实刻意没有使用
   - 有哪些不确定性或输入结构 fallback
   - 是否运行 lint，以及结果是什么

## 写作规则

### 从观众的困惑开始

不要用“今天我们来读一篇文章”开场。用具体钩子开场：

```text
你有没有发现……
一个很自然的问题来了……
听起来像是 A，但真正关键的是 B。
这件事表面上是 X，其实背后是 Y。
```

中国相关选题，通常先用中国生活经验开场，再引出外刊标签。优先采用：

```text
以前大家怎么理解这件事
现在画风哪里变了
外刊给了一个说法
但真正要解释的不是这个表面变化
```

不要一开头就列原文里的外来物品或猎奇对象，除非这些对象已经嵌在中文生活场景里。比如中国中产饮食这篇，不要用“三文鱼、蓝莓、牛油果越来越日常”开场；要先讲“大鱼大肉、请客有面子、年夜饭硬菜”这类中国人对“吃得好”的旧理解，然后再做反差。

可以在开头附近提一次来源：

```text
《经济学人》把这个变化称为……
《外交政策》这篇文章真正提醒的是……
```

之后就讲问题本身，不要反复说“文章说”。

### 把文章讲成模型

优先使用这个推进：

```text
表面现象 -> 主问题 -> 核心变量 -> 机制解释 -> 具体例子 -> 约束代价 -> 大判断
```

不要按原文段落顺序机械摘要。

趋势型或机制型文章，要尽早说出变量组。好的口播会让观众先拿到地图：

```text
这件事背后大概有四五个变量：钱、信任、健康、供应链、便利。
```

只有当原文真的支撑变量组时才这样写。不要给单一事件叙事或人物故事硬凑清单。

默认解释型文章里，变量组要出现在第一个深度背景段之前。观众要先知道地图，再进入细节。

机制重的文章，写作时问这些问题：

- 什么变了？
- 为什么偏偏是现在？
- 哪些变量在驱动？
- 谁受益，谁付钱，谁被限制？
- 最常见的解释漏掉了什么？
- 这件事揭示了更大系统里的什么问题？

系统重的文章，写作时问这些问题：

- 系统有哪些节点？
- 事情必须按什么顺序发生？
- 哪些地方容易失败？
- 什么让它难以扩张或治理？
- 这个系统揭示了权力、金钱、信任或协作里的什么问题？

### 使用问题链

每一节都要像一个干净的问答单元：

```text
问题：……
短答案：……
机制：……
例子：……
下一问：……
```

最终稿里不要写这些标签。它们只是写作时的内在结构。

### 保持单人口播友好

- 使用自然中文，不要翻译腔。
- 段落要短，方便人声和 TTS 换气。
- 在解释、场景、数字、例子之间切换，不要连续堆抽象判断。
- 每 1 到 2 分钟重新锚定观众一次：比如“到这里，我们先得到第一个结论……”
- 主持人可以在单人口播里自问自答：比如“那问题来了……”“为什么偏偏是现在？”“这是不是说明……？”然后直接回答。
- 不要使用 `Speaker 0:`、`Speaker 1:`、角色名或舞台指令。
- 不要编造轶事、引语或原文没有支撑的事实。

### 保留复杂性

稿子要清楚，但不能把复杂问题讲扁。

- 如果趋势真实但有限，要两边都说
- 如果政策有用但有代价，要两边都说
- 如果来源本身有不确定性，不要替它下过度结论
- 如果事实只适用于中产、城市、某个行业或某个时间段，不要扩大成所有人

## 质量门

完成后尽量运行 lint：

```bash
python3 /Users/wangfangjia/.codex/skills/article-single-host-explainer-script/scripts/lint_single_host_script.py <project>/single_host_script.md
```

检查重点：

```text
single_host_script.md 存在
planning/single_host_explainer_plan.json 存在
planning/single_host_script_generation_report.md 存在
稿件没有 Speaker 0/Speaker 1 标签
开头不是读书报告或文章摘要腔
主问题在全文前 20% 内清楚出现
章节按“提问，短答，机制，例子，下一问”推进
事实没有越过原文和策划文件边界
稿件包含反直觉澄清，也包含至少一个限制、代价或矛盾
```
